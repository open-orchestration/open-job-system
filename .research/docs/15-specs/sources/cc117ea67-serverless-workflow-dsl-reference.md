## Abstract

[Permalink: Abstract](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#abstract)

This document provides comprehensive definitions and detailed property tables for all the concepts discussed in the Serverless Workflow DSL. It serves as a reference guide, explaining the structure, components, and configurations available within the DSL. By exploring this document, users will gain a thorough understanding of how to define, configure, and manage workflows, including task definitions, flow directives, and state transitions. This foundational knowledge will enable users to effectively utilize the DSL for orchestrating serverless functions and automating processes.

## Definitions

[Permalink: Definitions](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#definitions)

### Workflow

[Permalink: Workflow](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#workflow)

A [workflow](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#workflow) serves as a blueprint outlining the series of [tasks](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#task) required to execute a specific business operation. It details the sequence in which [tasks](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#task) must be completed, guiding users through the process from start to finish, and helps streamline operations, ensure consistency, and optimize efficiency within an organization.

### Task

[Permalink: Task](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#task)

A task within a [workflow](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#workflow) represents a discrete unit of work that contributes to achieving the overall objectives defined by the [workflow](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#workflow).

It encapsulates a specific action or set of actions that need to be executed in a predefined order to advance the workflow towards its completion.

[Tasks](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#task) are designed to be modular and focused, each serving a distinct purpose within the broader context of the [workflow](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#workflow).

By breaking down the [workflow](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#workflow) into manageable [tasks](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#task), organizations can effectively coordinate and track progress, enabling efficient collaboration and ensuring that work is completed in a structured and organized manner.

The Serverless Workflow DSL defines a list of [tasks](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#task) that **must be** supported by all runtimes:

- [Call](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#call), used to call services and/or functions.
- [Do](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#do), used to define one or more subtasks to perform in sequence.
- [Fork](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#fork), used to define one or more subtasks to perform concurrently.
- [Emit](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#emit), used to emit [events](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#event-properties).
- [For](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#for), used to iterate over a collection of items, and conditionally perform a task for each of them.
- [Listen](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#listen), used to listen for an [event](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#event-properties) or more.
- [Raise](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#raise), used to raise an [error](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#error) and potentially fault the [workflow](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#workflow).
- [Run](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#run), used to run a [container](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#container-process), a [script](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#script-process) , a [shell](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#shell-process) command or even another [workflow](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#workflow-process).
- [Switch](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#switch), used to dynamically select and execute one of multiple alternative paths based on specified conditions
- [Set](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#set), used to dynamically set the [workflow](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#workflow)'s data during the its execution.
- [Try](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#try), used to attempt executing a specified [task](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#task), and to handle any resulting [errors](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#error) gracefully, allowing the [workflow](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#workflow) to continue without interruption.
- [Wait](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#wait), used to pause or wait for a specified duration before proceeding to the next task.

#### Properties

[Permalink: Properties](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#properties-1)

| Name | Type | Required | Description |
| :-- | :-: | :-: | :-- |
| if | `string` | `no` | A [`runtime expression`](https://github.com/serverlessworkflow/specification/blob/main/dsl.md#runtime-expressions), if any, used to determine whether or not the task should be run.<br>The task is considered skipped if not run, and the _raw_ task input becomes the task's output. The expression is evaluated against the _raw_ task input before any other expression of the task. |

| output | [`output`](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#output) | `no` | An object used to customize the task's output and to document its schema, if any. |
| export | [`export`](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#export) | `no` | An object used to customize the content of the workflow context. |
| timeout | `string`<br>[`timeout`](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#timeout) | `no` | The configuration of the task's timeout, if any.<br>_If a `string`, must be the name of a [timeout](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#timeout) defined in the [workflow's reusable components](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#use)._ |

| metadata | `map` | `no` | Additional information about the task. |

| method | `string` | `yes` | The A2A JSON-RPC method to send.<br>_Supported values are: `message/send`, `message/stream`, `tasks/get`, `tasks/list`, `tasks/cancel`, `tasks/resubscribe`, `tasks/pushNotificationConfig/set`, `tasks/pushNotificationConfig/get`, `tasks/pushNotificationConfig/list`, `tasks/pushNotificationConfig/delete`, and `agent/getAuthenticatedExtendedCard`_ |

| parameters | `map`<br>`string` | `no` | The parameters for the A2A RPC method. For the `message/send` and `message/stream` methods, runtimes must default `message.messageId` to a uuid and `message.role` to `user`.<br>_Supports [runtime expressions](https://github.com/serverlessworkflow/specification/blob/main/dsl.md#runtime-expressions)._ |

Note

#### Listen

[Permalink: Listen](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#listen)

Provides a mechanism for workflows to await and react to external events, enabling event-driven behavior within workflow systems.

##### Properties

[Permalink: Properties](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#properties-13)

| Name | Type | Required | Description |
| :-- | :-: | :-: | :-- |
| listen.to | [`eventConsumptionStrategy`](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#event-consumption-strategy) | `yes` | Configures the [event(s)](https://cloudevents.io/) the workflow must listen to. |
| listen.read | `string` | `no` | Specifies how [events](https://cloudevents.io/) are read during the listen operation.<br>_Supported values are:_<br>_\- `data`: Reads the [event's](https://cloudevents.io/) data._<br>_\- `envelope`: Reads the [event's](https://cloudevents.io/) envelope, including its [context attributes](https://github.com/cloudevents/spec/blob/main/cloudevents/spec.md#context-attributes)._<br>_\- `raw`: Reads the [event's](https://cloudevents.io/) raw data._<br>_Defaults to `data`._ |
| foreach | [`subscriptionIterator`](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#subscription-iterator) | `no` | Configures the iterator, if any, for processing each consumed [event](https://cloudevents.io/). |

Note

A `listen` task produces a sequentially ordered array of all the [events](https://cloudevents.io/) it has consumed, and potentially transformed using `foreach.output.as`.

Note

When `foreach` is set, the configured operations for a [events](https://cloudevents.io/) must complete before moving on to the next one. As a result, consumed [events](https://cloudevents.io/) should be stored in a First-In-First-Out (FIFO) queue while awaiting iteration.

Warning

[Events](https://cloudevents.io/) consumed by an `until` clause should not be included in the task's output. These [events](https://cloudevents.io/) are used solely to determine when the until condition has been met, and they do not contribute to the result or data produced by the task itself

##### Examples

[Permalink: Examples](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#examples-12)

```

  dsl: '1.0.3'
  namespace: test
  name: listen-example
  version: '0.1.0'
do:
  - callDoctor:
      listen:
        to:
          any:
          - with:
              type: com.fake-hospital.vitals.measurements.temperature
              data: ${ .temperature > 38 }
          - with:
              type: com.fake-hospital.vitals.measurements.bpm
              data: ${ .bpm < 60 or .bpm > 100 }
```

### Flow Directive

[Permalink: Flow Directive](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#flow-directive)

Flow Directives are commands within a workflow that dictate its progression.

| Directive | Description |
| --- | --- |
| `"continue"` | Instructs the workflow to proceed with the next task in line. This action may conclude the execution of a particular workflow or branch if there are not task defined after the continue one. |
| `"exit"` | Completes the current scope's execution, potentially terminating the entire workflow if the current task resides within the main `do` scope. |
| `"end"` | Provides a graceful conclusion to the workflow execution, signaling its completion explicitly. |
| `string` | Continues the workflow at the task with the specified name. |

Warning

Flow directives may only redirect to tasks declared within their own scope. In other words, they cannot target tasks at a different depth.

### Extension

[Permalink: Extension](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#extension)

Holds the definition for extending functionality, providing configuration options for how an extension extends and interacts with other components.

Extensions enable the execution of tasks prior to those they extend, offering the flexibility to potentially bypass the extended task entirely using an [`exit` workflow directive](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#flow-directive).

#### Properties

[Permalink: Properties](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#properties-52)

| Property | Type | Required | Description |
| --- | :-: | :-: | --- |
| extend | `string` | `yes` | The type of task to extend<br>Supported values are: `call`, `composite`, `emit`, `extension`, `for`, `listen`, `raise`, `run`, `set`, `switch`, `try`, `wait` and `all` |
| when | `string` | `no` | A runtime expression used to determine whether or not the extension should apply in the specified context |
| before | [`task[]`](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#task) | `no` | The list of tasks to execute, if any, before the extended task |
| after | [`task[]`](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#task) | `no` | The list of tasks to execute, if any, after the extended task |

#### Examples

[Permalink: Examples](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#examples-49)

_Perform logging before and after any non-extension task is run:_

```
document:
  dsl: '1.0.3'
  namespace: test
  name: logging-extension-example
  version: '0.1.0'
use:
  extensions:
    - logging:
        extend: all
        before:
          - sendLog:
              call: http
              with:
                method: post
                endpoint: https://fake.log.collector.com
                body:
                  message: ${ "Executing task '\($task.reference)'..." }
        after:
          - sendLog:
              call: http
              with:
                method: post
                endpoint: https://fake.log.collector.com
                body:
                  message: ${ "Executed task '\($task.reference)'..." }
do:
  - sampleTask:
      call: http
      with:
        method: get
        endpoint: https://fake.com/sample
```

_Intercept HTTP calls to ' [https://mocked.service.com](https://mocked.service.com/)' and mock its response:_

```
document:
  dsl: '1.0.3'
  namespace: test
  name: intercept-extension-example
  version: '0.1.0'
use:
  extensions:
    - mockService:
        extend: call
        when: $task.call == "http" and ($task.with.uri != null and ($task.with.uri | startswith("https://mocked.service.com"))) or ($task.with.endpoint.uri != null and ($task.with.endpoint.uri | startswith("https://mocked.service.com")))
        before:
          - intercept:
              set:
                statusCode: 200
                headers:
                  Content-Type: application/json
                content:
                  foo:
                    bar: baz
              then: exit #using this, we indicate to the workflow we want to exit the extended task, thus just returning what we injected
do:
```

### URI Template

[Permalink: URI Template](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#uri-template)

The DSL has limited support for URI template syntax as defined by [RFC 6570](https://datatracker.ietf.org/doc/html/rfc6570). Specifically, only the [Simple String Expansion](https://datatracker.ietf.org/doc/html/rfc6570#section-3.2.2) is supported, which allows authors to embed variables in a URI.

To substitute a variable within a URI, use the `{}` syntax. The identifier inside the curly braces will be replaced with its value during runtime evaluation. If no value is found for the identifier, an empty string will be used.

This has the following limitations compared to runtime expressions:

- Only top-level properties can be interpolated within strings, thus identifiers are treated verbatim. This means that `{pet.id}` will be replaced with the value of the `"pet.id"` property, not the value of the `id` property of the `pet` property.
- The referenced variable must be of type `string`, `number`, `boolean`, or `null`. If the variable is of a different type an error with type `https://https://serverlessworkflow.io/spec/1.0.0/errors/expression` and status `400` will be raised.
- [Runtime expression arguments](https://github.com/serverlessworkflow/specification/blob/main/dsl.md#runtime-expression-arguments) are not available for string substitution.

#### Examples

[Permalink: Examples](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#examples-60)

```
uri: https://petstore.swagger.io/v2/pet/{petId}
```

### AsyncAPI Subscription

[Permalink: AsyncAPI Subscription](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#asyncapi-subscription)

Configures a subscription to an AsyncAPI operation.

#### Properties

[Permalink: Properties](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#properties-73)

| Name | Type | Required | Description |
| :-- | :-: | :-: | :-- |
| filter | `string` | `no` | A [runtime expression](https://github.com/serverlessworkflow/specification/blob/main/dsl.md#runtime-expressions), if any, used to filter consumed [messages](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#asyncapi-inbound-message). |
| consume | [`subscriptionLifetime`](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#asyncapi-subscription-lifetime) | `yes` | An object used to configure the subscription's lifetime. |
| foreach | [`subscriptionIterator`](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#subscription-iterator) | `no` | Configures the iterator, if any, for processing each consumed [message](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#asyncapi-inbound-message). |

Note

An AsyncAPI subscribe operation call produces a sequentially ordered array of all the [messages](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#asyncapi-inbound-message) it has consumed, and potentially transformed using `foreach.output.as`.

Note

When `foreach` is set, the configured operations for a [message](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#asyncapi-inbound-message) must complete before moving on to the next one. As a result, consumed [messages](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#asyncapi-inbound-message) should be stored in a First-In-First-Out (FIFO) queue while awaiting iteration.

#### Examples

[Permalink: Examples](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#examples-65)

```
document:
  dsl: '1.0.3'
  namespace: test
  name: asyncapi-example
  version: '0.1.0'
do:
  - subscribeToChatInboxForAmount:
      call: asyncapi
      with:
        document:
          endpoint: https://fake.com/docs/asyncapi.json
        operation: chat-inbox
        protocol: http
        subscription:
          filter: ${ . == $workflow.input.chat.roomId }
          consume:
            amount: 5
            for:
              seconds: 10
```

### Subscription Iterator

[Permalink: Subscription Iterator](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#subscription-iterator)

[Permalink: Examples](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md#examples-67)

```
        subscription:
          filter: ${ . == $workflow.input.chat.roomId }
          consume:

              seconds: 10
          foreach:
            item: message
            at: index
            do:

                  emit:
                    event:
                      with:

                          message: '${ $message }'
```
