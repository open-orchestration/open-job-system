# Triggers

When you push code to your repository, open a pull request, or create a tag, your source control management system automatically sends a webhook to Drone which in turn triggers pipeline execution. The trigger section is common across all pipeline types, and is used to limit pipeline execution.

Example limits pipeline execution by branch:

|     |     |
| --- | --- |
| ```<br> 1<br> 2<br> 3<br> 4<br> 5<br> 6<br> 7<br> 8<br> 9<br>10<br>11<br>12<br>13<br>14<br>``` | ```yaml<br>kind: pipeline<br>type: docker<br>name: default<br>steps:<br>- name: build<br>  image: golang<br>  commands:<br>  - go build<br>  - go test<br>trigger:<br>  branch:<br>  - master<br>``` |

You can use wildcard matching in your triggers. _Note that triggers use glob pattern matching, not regular expressions._

|     |     |
| --- | --- |
| ```<br>12<br>13<br>14<br>15<br>16<br>``` | ```yaml<br>trigger:<br>  ref:<br>  - refs/heads/master<br>  - refs/heads/**<br>  - refs/pull/*/head<br>``` |

You can also combine multiple triggers. _Note that all triggers must evaluate to true when combining multiple triggers._

|     |     |
| --- | --- |
| ```<br>12<br>13<br>14<br>15<br>16<br>``` | ```yaml<br>trigger:<br>  branch:<br>  - master<br>  event:<br>  - push<br>``` |

# By Branch

The branch trigger limits step execution based on the git branch. Please note that the target branch is evaluated for pull requests; and branch names are not available for tag events.

Note that you cannot use branch triggers with tags. A tag is not associated with the source branch from which it was created.

|     |     |
| --- | --- |
| ```<br>12<br>13<br>14<br>15<br>``` | ```yaml<br>trigger:<br>  branch:<br>  - master<br>  - feature/*<br>``` |

Example include syntax:

|     |     |
| --- | --- |
| ```<br>12<br>13<br>14<br>15<br>16<br>``` | ```yaml<br>trigger:<br>  branch:<br>    include:<br>    - master<br>    - feature/*<br>``` |

Example exclude syntax:

|     |     |
| --- | --- |
| ```<br>12<br>13<br>14<br>15<br>16<br>``` | ```yaml<br>trigger:<br>  branch:<br>    exclude:<br>    - master<br>    - feature/*<br>``` |

# By Event

The event trigger limits step execution based on the drone event type. This can be helpful when you want to limit steps based on push, pull request, tag and more.

Note that you cannot use branch triggers with tag events. A tag is not associated with the source branch from which it was created.

|     |     |
| --- | --- |
| ```<br>12<br>13<br>14<br>15<br>16<br>17<br>18<br>19<br>20<br>``` | ```yaml<br>trigger:<br>  event:<br>  - cron<br>  - custom<br>  - push<br>  - pull_request<br>  - tag<br>  - promote<br>  - rollback<br>``` |

Example include syntax:

|     |     |
| --- | --- |
| ```<br>12<br>13<br>14<br>15<br>16<br>``` | ```yaml<br>trigger:<br>  event:<br>    include:<br>    - push<br>    - pull_request<br>``` |

Example exclude syntax:

|     |     |
| --- | --- |
| ```<br>12<br>13<br>14<br>15<br>``` | ```yaml<br>trigger:<br>  event:<br>    exclude:<br>    - pull_request<br>``` |

# By Reference

The reference trigger limits step execution based on the git reference name. This can be helpful when you want to glob match branch or tag names.

|     |     |
| --- | --- |
| ```<br>12<br>13<br>14<br>15<br>``` | ```yaml<br>trigger:<br>  ref:<br>  - refs/heads/feature-*<br>  - refs/tags/*<br>``` |

Example include syntax:

|     |     |
| --- | --- |
| ```<br>12<br>13<br>14<br>15<br>16<br>17<br>``` | ```yaml<br>trigger:<br>  ref:<br>    include:<br>    - refs/heads/feature-*<br>    - refs/pull/**<br>    - refs/tags/**<br>``` |

Example exclude syntax:

|     |     |
| --- | --- |
| ```<br>12<br>13<br>14<br>15<br>16<br>17<br>``` | ```yaml<br>trigger:<br>  ref:<br>    exclude:<br>    - refs/heads/feature-*<br>    - refs/pull/**<br>    - refs/tags/**<br>``` |

# By Repository

The repository trigger limits step execution based on repository name. This can be useful when Drone is enabled for a repository and its forks, and you want to limit execution accordingly.

|     |     |
| --- | --- |
| ```<br>12<br>13<br>14<br>``` | ```yaml<br>trigger:<br>  repo:<br>  - octocat/hello-world<br>``` |

Example include syntax:

|     |     |
| --- | --- |
| ```<br>12<br>13<br>14<br>15<br>16<br>``` | ```yaml<br>trigger:<br>  repo:<br>    include:<br>    - octocat/hello-world<br>    - spacebhost/hello-world<br>``` |

Example exclude syntax:

|     |     |
| --- | --- |
| ```<br>12<br>13<br>14<br>15<br>16<br>``` | ```yaml<br>trigger:<br>  repo:<br>    exclude:<br>    - octocat/hello-world<br>    - spacebhost/hello-world<br>``` |

Example using wildcard matching:

|     |     |
| --- | --- |
| ```<br>12<br>13<br>14<br>15<br>``` | ```yaml<br>trigger:<br>  repo:<br>    include:<br>    - octocat/*<br>``` |

# By Status

The status trigger limits step execution based on the pipeline status. For example, you may want to configure Slack notification only on failure.

|     |     |
| --- | --- |
| ```<br>12<br>13<br>14<br>``` | ```yaml<br>trigger:<br>  status:<br>  - failure<br>``` |

Execute a step on failure:

|     |     |
| --- | --- |
| ```<br>12<br>13<br>14<br>``` | ```yaml<br>trigger:<br>  status:<br>  - failure<br>``` |

Execute a step on success or failure:

|     |     |
| --- | --- |
| ```<br>12<br>13<br>14<br>15<br>``` | ```yaml<br>trigger:<br>  status:<br>  - success<br>  - failure<br>``` |

The following configuration is redundant. The default behavior is for pipeline steps to only execute when the pipeline is in a passing state.

|     |     |
| --- | --- |
| ```<br>12<br>13<br>14<br>``` | ```yaml<br>trigger:<br>  status:<br>  - success<br>``` |

# By Target

The target trigger limits step execution based on the target deployment environment. This only applies to promotion and rollback events.

|     |     |
| --- | --- |
| ```<br>12<br>13<br>14<br>``` | ```yaml<br>trigger:<br>  target:<br>  - production<br>``` |

Example include syntax:

|     |     |
| --- | --- |
| ```<br>12<br>13<br>14<br>15<br>16<br>``` | ```yaml<br>trigger:<br>  target:<br>    include:<br>    - staging<br>    - production<br>``` |

Example exclude syntax:

|     |     |
| --- | --- |
| ```<br>12<br>13<br>14<br>15<br>``` | ```yaml<br>trigger:<br>  target:<br>    exclude:<br>    - production<br>``` |

# By Cron

The cron trigger limits step execution based on the cron name that triggered the pipeline. This only applies to cron events.

|     |     |
| --- | --- |
| ```<br>12<br>13<br>14<br>15<br>16<br>``` | ```yaml<br>trigger:<br>  event:<br>  - cron<br>  cron:<br>  - nightly<br>``` |

Example include syntax:

|     |     |
| --- | --- |
| ```<br>12<br>13<br>14<br>15<br>16<br>17<br>18<br>``` | ```yaml<br>trigger:<br>  event:<br>  - cron<br>  cron:<br>    include:<br>    - weekly<br>    - nightly<br>``` |

Example exclude syntax:

|     |     |
| --- | --- |
| ```<br>12<br>13<br>14<br>15<br>16<br>17<br>``` | ```yaml<br>trigger:<br>  event:<br>  - cron<br>  cron:<br>    exclude:<br>    - nightly<br>``` |

# By Action

The action trigger limits execution based on the action associated with the event that triggered the pipeline.

Action support varies by event and SCM provider.

Execute when a pull request is opened:

|     |     |
| --- | --- |
| ```<br>12<br>13<br>14<br>15<br>16<br>``` | ```yaml<br>trigger:<br>  event:<br>  - pull_request<br>  action:<br>  - opened<br>``` |

Execute on all pull request actions except “synchronized”:

|     |     |
| --- | --- |
| ```<br>12<br>13<br>14<br>15<br>16<br>17<br>``` | ```yaml<br>trigger:<br>  event:<br>  - pull_request<br>  action:<br>    exclude:<br>    - synchronized<br>``` |
