# GCP and Firebase Job and Serverless Execution Services: Cloud Run Jobs, Composer, Cloud Functions, and Extensions

Survey of four Google Cloud / Firebase services for running jobs and serverless code, each described strictly from its own source body. No cross-attribution: every claim carries only its own service's cid.

## Cloud Run jobs

Unlike a Cloud Run service, which listens for and serves requests, a Cloud Run job only runs its tasks and exits when finished; a job does not listen for or serve requests [c45b51242]. A job can be executed as a one-off, on a schedule, or as part of a workflow [c45b51242]. A job can be structured as a single task or as multiple, independent tasks (up to 10,000 tasks) that can be executed in parallel [c45b51242]. Each task runs one container instance and can be configured to retry in case of failure [c45b51242]. Each task is aware of its index, stored in the `CLOUD_RUN_TASK_INDEX` environment variable, and the overall count of tasks is stored in the `CLOUD_RUN_TASK_COUNT` environment variable [c45b51242]. When processing data in parallel, the user's code is responsible for determining which task handles which portion of the data [c45b51242].

The `--tasks` option accepts integers greater than or equal to 1, defaults to 1, and has a maximum of 10,000; each task is provided `CLOUD_RUN_TASK_INDEX` with a value between 0 and the number of tasks minus 1, along with `CLOUD_RUN_TASK_COUNT` [c45b51242]. The `--max-retries` option sets the number of times a failed task is retried; once any task fails beyond this limit, the entire job is marked as failed, and the default is 3 (accepting integers from 0 to 10) [c45b51242]. The `--task-timeout` option accepts a duration, defaults to 10 minutes, and has a maximum of 168 hours (7 days) [c45b51242]. The `--parallelism` option sets the maximum number of tasks that can execute in parallel; by default tasks are started as quickly as possible in parallel [c45b51242]. Jobs are created from an existing container image and can be created via the Console, gcloud, YAML, Terraform, client libraries, or the REST API [c45b51242].

## Cloud Composer

Managed Airflow is a fully managed workflow orchestration service, enabling the user to create, schedule, monitor, and manage workflow pipelines that span across clouds and on-premises data centers [cd1a4f815]. It is built on the Apache Airflow open source project and operates using the Python programming language [cd1a4f815]. By using Managed Airflow instead of a local instance of Apache Airflow, the user benefits from Airflow with no installation or management overhead and can use Airflow-native tools such as the Airflow web interface and command-line tools, focusing on workflows rather than infrastructure [cd1a4f815]. Cloud Composer is evolving to become Managed Service for Apache Airflow [cd1a4f815].

In Airflow, workflows are created using DAGs (Directed Acyclic Graphs); a DAG is a collection of tasks to schedule and run, organized to reflect their relationships and dependencies, created in Python files that define the DAG structure using code [cd1a4f815]. The DAG's purpose is to ensure each task executes at the right time and in the right order; each task can represent almost anything, such as preparing data for ingestion, monitoring an API, sending an email, or running a pipeline [cd1a4f815]. In addition to running a DAG on a schedule, DAGs can be triggered manually or in response to events such as changes in a Cloud Storage bucket [cd1a4f815].

Managed Airflow environments are self-contained Airflow deployments based on Google Kubernetes Engine, working with other Google Cloud services through connectors built into Airflow; one or more environments can be created in a single project, in any supported region [cd1a4f815]. Managed Airflow provisions the Google Cloud services that run the workflows and all Airflow components: a GKE cluster where Airflow schedulers, triggerers, and workers run as GKE workloads to process and execute DAGs; an Airflow web server running the Apache Airflow UI; an Airflow database holding the Apache Airflow metadata; and a Cloud Storage bucket (the environment's bucket) storing the DAGs, logs, custom plugins, and data [cd1a4f815]. Managed Airflow lets the user manage Airflow features including DAGs, Airflow configuration options, connections, the Airflow UI, the Airflow CLI, custom plugins, Python dependencies, and logging and monitoring for DAGs and components [cd1a4f815]. Managed Airflow (Gen 3) supports Airflow 2, Gen 2 supports Airflow 2, and Legacy Gen 1 supports Airflow 1 and Airflow 2 [cd1a4f815].

## Cloud Functions for Firebase

Cloud Functions for Firebase is a serverless framework that lets the user automatically run backend code in response to events triggered by background events, HTTPS requests, the Admin SDK, or Cloud Scheduler jobs [c32a79a8f]. JavaScript, TypeScript, or Python code is stored on Google Cloud infrastructure and runs in a managed environment, with no need to manage and scale one's own servers [c32a79a8f]. There is experimental support for Dart, where during the preview Dart support is available for HTTP and callable functions [c32a79a8f].

The functions written can respond to events generated by various Firebase and Google Cloud features, from Firebase Authentication triggers to Cloud Storage Triggers, and can integrate across Firebase features using the Admin SDK and integrate with third-party services via webhooks [c32a79a8f]. Cloud Functions minimizes boilerplate code [c32a79a8f]. For the zero-maintenance capability, the user deploys JavaScript, TypeScript, or Python code with one command from the command line, after which Firebase automatically scales up computing resources to match users' usage patterns, with no need to worry about credentials, server configuration, provisioning new servers, or decommissioning old ones [c32a79a8f]. Cloud Functions is fully insulated from the client, keeping the logic private and secure [c32a79a8f].

After a function is written and deployed, Google's servers begin to manage it immediately; the function can be fired directly with an HTTP request, the Admin SDK, or a scheduled job, and for background functions Google's servers listen for events and run the function when it is triggered [c32a79a8f]. As load increases or decreases, Google responds by rapidly scaling the number of virtual server instances needed to run the function, and each function runs in isolation in its own environment with its own configuration [c32a79a8f].

## Firebase Extensions

A Firebase Extension is code that performs a task whenever a specifically defined event occurs in the app or project [c812a53be]. An extension is a packaged solution, so the user only installs and configures the extension rather than researching, writing, and debugging the code that implements the functionality [c812a53be]. Each installed instance of an extension is unique: the user specifies configuration values unique to the app, project, or use case (such as a Cloud Firestore path, image dimensions, or a GitHub URL) [c812a53be]. The same extension can be reused in different projects, and multiple instances of the same extension can be installed in a single project, each with its own customized configuration [c812a53be].

An extension's logic is written using Cloud Functions for Firebase; the functions in an extension define the event providers and the conditions that trigger execution (for example, a Cloud Firestore write, an HTTPS request, or a Cloud Scheduler event) [c812a53be]. A key differentiating feature between extensions and functions is that an extension relies on an `extension.yaml` specification file, which explicitly defines the Google services (APIs) the extension will use, the access roles it requires, the extension-specific resources it requires, and its configurable parameters [c812a53be]. Extensions can respond to events generated by Firebase products already in use — a change in one Firebase product can trigger an extension to perform a task using another product (for example, a Realtime Database write triggering a Firebase Cloud Messaging notification) — and can integrate a Firebase project with other Google Cloud Platform products (such as BigQuery and Google Translate) or third-party services (such as Mailchimp and Bit.ly); triggering events are not limited to Firebase events, as an extension can be triggered directly with an HTTPS request or at a scheduled interval [c812a53be].

When an instance is installed, Firebase prompts for the required configuration values (parameters), enables the listed APIs from the `extension.yaml` file, creates a new service account assigned the listed access roles under which the instance executes its code, and provisions the listed resources (for example, a function) [c812a53be]. Each installed instance has its own service account and individually-provisioned resources [c812a53be]. After installation, an extension can be reconfigured with new parameter values, updated to a newer version, or uninstalled at any time, all via the Firebase CLI or the Firebase console [c812a53be]. The application logic for extensions runs on the backend using Google Cloud Functions, fully isolated from the client, and an installed extension is granted limited access only to the resources and data clearly listed before installation [c812a53be]. For the zero-maintenance capability, after install and configuration the backend automatically scales computing resources up and down to match the extension's needs, with no worry about credentials, server configuration, or provisioning and decommissioning servers [c812a53be].
