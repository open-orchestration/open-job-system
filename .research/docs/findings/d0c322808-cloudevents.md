# CloudEvents: the event envelope, JSON format, and protocol bindings

## 1. The CloudEvents idea

The explicit goal of the CloudEvents effort, as expressed in the specification, is "describing event data in a common way" and "to define interoperability of event systems that allow services to produce or consume events, where the producer and consumer can be developed and deployed independently" [c2b839970]. The foundations for such interoperability are open data formats and open protocols, with CloudEvents aiming to provide such an open data format and projections of its data format onto commonly used protocols and with commonly used encodings [c2b839970]. CloudEvents is a standardized and protocol-agnostic definition of the structure and metadata description of events [c2c73f13a].

The motivation is concrete: without a common way of describing events, delivery of events across environments is hindered, and there is no single way of determining where an event came from and where it might be going [c2b839970]. CloudEvents offers useful metadata which middleware and consumers can rely upon to facilitate event routing, logging, delivery and receipt [c2b839970]. Its common way of describing event data also increases the portability of Functions-as-a-Service, a domain where vendor lock-in is partially caused by differences in the format of event data received within functions [c2b839970].

Conceptually, an event includes context and data about an occurrence, and each occurrence is uniquely identified by the data of the event [c2b839970]. Events represent facts and therefore do not include a destination, whereas messages convey intent, transporting data from a source to a given destination [c2b839970].

### Context attributes

The JSON format's exemplary attribute-mapping table enumerates the core context attributes with their types and example values: `type` (String, e.g. `"com.example.someevent"`), `specversion` (String, e.g. `"1.0"`), `source` (URI-reference, e.g. `"/mycontext"`), `subject` (String, e.g. `"larger-context"`), `id` (String, e.g. `"1234-1234-1234"`), `time` (Timestamp, e.g. `"2018-04-05T17:31:00Z"`), and `datacontenttype` (String, e.g. `"application/json"`) [c2c73f13a]. The same table shows that `subject` and `time` may carry a `null` value [c2c73f13a].

On the `id` attribute specifically, the primer states that `id` is meant to be a value that is unique across all events related to one event source (where each event source is uniquely identified by its CloudEvents `source` attribute value); while the exact value used is producer defined, receivers of CloudEvents from a single event source can be assured that no two events will share the same `id` value [c2b839970]. Since a single occurrence may result in the generation of more than one event, in the cases where all of those events are from the same event source, each CloudEvent constructed will have a unique `id` [c2b839970].

## 2. The JSON event format envelope

The JSON Event Format specification defines how the elements defined in the CloudEvents specification are to be represented in the JavaScript Object Notation (JSON) Data Interchange Format [c2c73f13a]. Each CloudEvents event can be wholly represented as a JSON object, and such a representation MUST use the media type `application/cloudevents+json` [c2c73f13a]. All REQUIRED and all not omitted OPTIONAL attributes in the given event MUST become members of the JSON object, with the respective JSON object member name matching the attribute name, and the member's type and value being mapped using the type system mapping; OPTIONAL not omitted attributes MAY be represented as a `null` JSON value [c2c73f13a].

The CloudEvents type system MUST be mapped to JSON types as follows: Boolean to JSON boolean; Integer to JSON number (only the integer component, optionally prefixed with a minus sign); String to JSON string; Binary to a Base64-encoded JSON string; URI and URI-reference to JSON strings following RFC 3986; and Timestamp to a JSON string following RFC 3339 (ISO 8601) [c2c73f13a]. Unset attributes MAY be encoded to the JSON value of `null`, and when decoding attributes a `null` value MUST be treated as the equivalent of unset or omitted [c2c73f13a].

For the `data` member, a JSON serializer MUST first determine the runtime data type of the `data` content [c2c73f13a]. If the implementation determines that the type of data is `Binary`, the value MUST be represented as a JSON string containing the Base64 encoded binary value, stored under the member name `data_base64` [c2c73f13a]. The specification notes that the definition of `data_base64` is a JSON-specific marshaling rule and not part of the formal CloudEvents context attributes definition [c2c73f13a]. On deserialization, the presence of the `data_base64` member clearly indicates that the value is Base64 encoded binary data, which the deserializer MUST decode into a binary runtime data type [c2c73f13a].

The JSON format also defines a JSON Batch Format whose members are CloudEvents, with a corresponding batch envelope [c2c73f13a].

## 3. The HTTP protocol binding

The CloudEvents specification defines three content modes for transferring events: structured, binary and batch, and the HTTP protocol binding supports all three content modes, with every compliant implementation expected to support both structured and binary modes [c20f15872].

In the binary content mode, the value of the event `data` is placed into the HTTP request, or response, body as-is, with the `datacontenttype` attribute value declaring its media type in the HTTP `Content-Type` header; all other event attributes are mapped to HTTP headers [c20f15872]. Except where noted, all CloudEvents context attributes, including extensions, MUST be mapped to HTTP headers with the same name as the attribute name but prefixed with `ce-` — for example `time` maps to `ce-time`, `id` maps to `ce-id`, and `specversion` maps to `ce-specversion` [c20f15872]. For the binary mode, the HTTP `Content-Type` header value corresponds to the CloudEvents `datacontenttype` attribute, and a `ce-datacontenttype` HTTP header MUST NOT also be present in the message [c20f15872]. A binary-mode HTTP POST therefore carries headers such as `ce-specversion: 1.0`, `ce-type: com.example.someevent`, `ce-id: 1234-1234-1234`, and `ce-source: /mycontext/subcontext`, with `Content-Type: application/json` and the application data in the body [c20f15872].

In the structured content mode, event metadata attributes and event data are placed into the HTTP request or response body using an event format that supports structured-mode messages [c20f15872]. The HTTP `Content-Type` header MUST be set to the media type of an event format — for the JSON format this is `Content-Type: application/cloudevents+json; charset=UTF-8` [c20f15872]. The chosen event format defines how all attributes, and `data`, are represented, and the event metadata and data is then rendered in accordance with the event format specification, the resulting data becoming the HTTP message body [c20f15872].

In the batched content mode, event metadata attributes and event data of multiple events are batched into a single HTTP request or response body using an event format that supports batching structured-mode messages [c20f15872]. The HTTP `Content-Type` header MUST be set to the media type of the batch mode for the event format — for the JSON Batch format this is `Content-Type: application/cloudevents-batch+json; charset=UTF-8` — and the body is a JSON array of event objects [c20f15872].

The receiver distinguishes between the three modes by inspecting the `Content-Type` header value: if the value is prefixed with the CloudEvents media type `application/cloudevents` (matched case-insensitively) the receiver uses structured mode; if it is prefixed with `application/cloudevents-batch` the receiver uses batched mode; otherwise it defaults to binary mode [c20f15872]. The batched mode MUST NOT be used unless solicited, and the gesture SHOULD allow the receiver to choose the maximum size of a batch [c20f15872].

## 4. The Kafka protocol binding

The CloudEvents specification defines three content modes for transferring events — structured, binary and batch — but the Kafka protocol binding does not currently support the batch content mode, while every compliant implementation should support both structured and binary modes [cb37a61c6]. Implementations that use Kafka 0.11.0.0 and above MAY use either binary or structured modes; implementations that use Kafka 0.10.x.x and below MUST only use structured mode, because older versions of Kafka lacked support for message level headers [cb37a61c6].

For the Kafka key, every implementation MUST, by default, map the user provided record key to the Kafka record key [cb37a61c6]. The 'key' of the Kafka message MAY be populated by a "Key Mapper" function, and every implementation SHOULD provide an opt-in "Key Mapper" implementation that maps the Partitioning `partitionkey` attribute value to the 'key' of the Kafka message as-is, if present [cb37a61c6]. A mapping function MUST NOT modify the CloudEvent, so the `partitionkey` attribute MUST still be included with the transmitted event if present [cb37a61c6].

In the binary content mode, the value of the event `data` MUST be placed into the Kafka message's value section as-is, with the `content-type` header value declaring its media type, and all other event attributes MUST be mapped to the Kafka message's header section [cb37a61c6]. All CloudEvents attributes and extensions with the exception of `data` MUST be individually mapped to and from the Header fields in the Kafka message, and both header keys and header values MUST be encoded as UTF-8 strings [cb37a61c6]. CloudEvent attributes are prefixed with `ce_` for use in the message-headers section — for example `time` maps to `ce_time`, `id` maps to `ce_id`, and `specversion` maps to `ce_specversion` [cb37a61c6]. A binary-mode Kafka message thus carries a key (e.g. `Key: mykey`), headers such as `ce_specversion: "1.0"`, `ce_type: "com.example.someevent"`, `ce_source: "/mycontext/subcontext"`, `ce_id: "1234-1234-1234"`, and `ce_time`, a `content-type` header for the payload data (e.g. `application/avro`), and the application data in the value section [cb37a61c6].

In the structured content mode, event metadata attributes and event data are placed into the Kafka message value section using an event format [cb37a61c6]. If present, the Kafka message header property `content-type` MUST be set to the media type of an event format — for the JSON format, `content-type: application/cloudevents+json; charset=UTF-8` [cb37a61c6]. The chosen event format defines how all attributes, and `data`, are represented, and the event metadata and data are rendered in accordance with the event format specification, the result becoming the Kafka application data (value) section [cb37a61c6]. In structured mode, the Kafka representation of a CloudEvent with no `data` is a Kafka message which still has a data section containing the attributes of the CloudEvent, and such a message does not represent a tombstone record in a topic with log compaction enabled, unlike the representation in binary mode [cb37a61c6].

On the relationship between `data` and `datacontenttype` in Kafka, `data` is assumed to contain opaque application data that is encoded as declared by the `datacontenttype` attribute; for instance, if the declared `datacontenttype` is `application/json;charset=utf-8`, the expectation is that the `data` value is made available as UTF-8 encoded JSON text [cb37a61c6].
