# Temporal production hardening: security, payload codecs, and SDK testing

Taking a Temporal deployment to production raises three distinct concerns that the official documentation treats separately: securing the Temporal Service itself, encrypting the payloads it persists, and testing Workflow code deterministically. This finding synthesizes the Platform security guide, the Codecs and Encryption guide, and the Go SDK testing-suite guide into one production-hardening view.

## 1. Security: encryption in transit, authentication, and authorization

A secured Temporal Server has its network communication encrypted and has authentication and authorization protocols set up for API calls made to it; without these, your server could be accessed by unwanted entities [caa7de4d5]. The measures documented are built-in opt-in security measures that come with Temporal, though users may also choose to design their own security architecture with reverse proxies or run unsecured instances inside of a VPC environment [caa7de4d5].

### Encryption in transit with mTLS

Temporal supports Mutual Transport Layer Security (mTLS) as a way of encrypting network traffic between the services of a Temporal Service and also between application processes and a Temporal Service [caa7de4d5]. Self-signed or properly minted certificates can be used for mTLS [caa7de4d5]. The TLS configuration includes two sections such that intra-Temporal Service and external traffic can be encrypted with different sets of certificates and settings: `internode`, configuration for encrypting communication between nodes in the Temporal Service, and `frontend`, configuration for encrypting the Frontend's public endpoints [caa7de4d5]. A customized configuration can be passed using either the `WithConfig` or `WithConfigLoader` Server options [caa7de4d5].

### Authentication

There are a few authentication protocols available to prevent unwanted access such as authentication of servers, clients, and users [caa7de4d5]. To prevent spoofing and MITM attacks you can specify the `serverName` in the `client` section of your respective mTLS configuration; this enables established connections to authenticate the endpoint, ensuring that the server certificate presented to any connecting Client has the appropriate server name in its CN property, and it can be used for both `internode` and `frontend` endpoints [caa7de4d5]. To restrict a client's network access to Temporal Service endpoints you can limit it to clients with certificates issued by a specific Certificate Authority (CA), using the `clientCAFiles`/`clientCAData` and `requireClientAuth` properties in both the `internode` and `frontend` sections of the mTLS configuration [caa7de4d5].

### Authorization

Temporal offers two plugin interfaces for implementing API call authorization: `ClaimMapper` and `Authorizer` [caa7de4d5]. When these are provided the frontend invokes the implementation of these interfaces before executing the requested operation [caa7de4d5]. The Claim Mapper component is a pluggable component that extracts Claims from JSON Web Tokens (JWTs), achieved with the method `GetClaims`, which translates `AuthInfo` structs from the caller into `Claims` about the caller's roles within Temporal [caa7de4d5]. A `Role` within Temporal is a bit mask that combines one or more of the role constants [caa7de4d5]. The `Authorizer` plugin contains a single `Authorize` method, which is invoked for each incoming API call and receives information about the API call along with the role and permission claims of the caller; it then returns one of two decisions: `DecisionDeny`, where the requested API call is not invoked and an error is returned to the caller, or `DecisionAllow`, where the requested API call is invoked [caa7de4d5].

The default posture is dangerous: if you do not explicitly configure an `Authorizer`, Temporal uses the default `noopAuthorizer`, which allows every API request with no authentication or access control, so anyone who can reach your Temporal Server can invoke any API, including sensitive administrative operations [caa7de4d5]. To protect your Temporal Server, you must configure an `Authorizer` plugin with a corresponding `ClaimMapper`; without this, your deployment is effectively open to anyone with network access [caa7de4d5]. You configure your `Authorizer` with the `temporal.WithAuthorizer` server option and your `ClaimMapper` with the `temporal.WithClaimMapper` server option [caa7de4d5].

For SDK callers, when authentication is enabled the Temporal Server expects an `authorization` gRPC header with an authorization token to be passed with API calls if requests authorization is configured [caa7de4d5]. The default JWT `ClaimMapper` expects authorization tokens to be formatted as `Bearer <token>`, and each Individual Permission Claim must be formatted as `<namespace> : <permission>`, which are converted into one of Temporal's four values: read, write, worker, or admin [caa7de4d5].

## 2. Payload data encryption: Codecs and the Codec Server

The Temporal Service persists data from your Workflow Executions, including inputs, outputs, and results; to protect sensitive data, use a Payload Codec to encrypt payloads before they reach the Temporal Service [c89d6b902]. With encryption enabled, data exists unencrypted only on the Client and the Worker process, on hosts that you control [c89d6b902]. The data persisted in the Event History that can be encrypted includes inputs and outputs/results in your Workflow, Activity, and Child Workflow; Signal inputs; Memo; Query inputs and results; and results of Local Activities and Side Effects [c89d6b902]. Failure messages and call stacks are not encoded as codec-capable Payloads by default; you must explicitly enable encoding these common attributes on failures [c89d6b902].

### The Codec Server

To view encrypted data in the Web UI and CLI, set up a Codec Server [c89d6b902]. A Codec Server is an HTTP server that uses your custom Codec logic to decode your data remotely; it is independent of the Temporal Service and decodes your encrypted payloads through predefined endpoints, and you create, operate, and manage access to your Codec Server in your own environment [c89d6b902]. When you configure a Codec Server endpoint in the Web UI or CLI, the Web UI and CLI use the remote endpoint to send and receive payloads from the Codec Server [c89d6b902]. When you use a Codec Server, the decoded payloads are decoded and returned on the client side only, and payloads on the Temporal Service (whether on Temporal Cloud or self-hosted) remain encrypted [c89d6b902]. Your Codec Server should share logic with the custom Payload Codec used elsewhere in your application [c89d6b902].

The Web UI and CLI send POST requests to the following endpoints on your Codec Server: `/decode` passes incoming payloads to the decode method in your Payload Codec; `/encode` passes incoming payloads to the encode method in your Payload Codec; and `/download` retrieves and decodes payloads from External Storage, an endpoint only needed if your Workers use External Storage [c89d6b902]. Each request from the Web UI includes a `Content-Type: application/json` header and an `X-Namespace: {namespace}` custom HTTP header, with an optional `Authorization: <credentials>` header when enabling authorization [c89d6b902]. By default, all field values in your payload are base64 encoded, regardless of whether they are encrypted by your custom codec implementation [c89d6b902]. You will need to enable CORS on the HTTP/HTTPS endpoints in your Codec Server to receive requests from the Temporal Web UI [c89d6b902].

Because the Codec Server is designed to decode potentially sensitive data with a single API call, access to a production Codec Server should be restricted [c89d6b902]. Depending on your infrastructure and risk levels, it might be sufficient to restrict HTTP ingress to your Codec Server, such as by using a VPN, and the Temporal Web UI can communicate with a Codec Server that is only accessible on `localhost`, so this is a legitimate security pattern; however, if your Codec Server is exposed to the internet at all, you will likely need an authentication solution [c89d6b902]. To enable authorization from the Web UI, your Codec Server must be an HTTPS Server [c89d6b902]. You can configure a Codec Server endpoint with the Temporal CLI using the `--codec-endpoint` flag, and if your Codec Server requires authentication, the Temporal CLI will also accept a `--codec-auth` parameter to supply an authorization header [c89d6b902].

### Large-payload handling

If your payloads exceed the Temporal Service's size limits, use External Storage to offload large payloads to an external store like Amazon S3 [c89d6b902]. The Web UI displays a claim reference instead of the payload content for externally stored payloads, and when External Storage is configured on your Codec Server, the Codec Server can retrieve and decode these payloads for viewing in the Web UI and CLI [c89d6b902].

## 3. Testing: the Temporal Go SDK test framework

The Temporal Go SDK provides a test framework to facilitate testing Workflow implementations, suited for implementing unit tests as well as functional tests of the Workflow logic [cf4aa1182]. You can create end-to-end tests (running a Temporal Server and Worker with all its Workflows and Activities), integration tests (anything between end-to-end and unit testing), and unit tests (running a piece of Workflow or Activity code and mocking any code it calls); the guidance is to write the majority of your tests as integration tests [cf4aa1182]. Because the test server supports skipping time, you use the test server for both end-to-end and integration tests with Workers [cf4aa1182].

### Test environment setup

To run unit tests, you first define a test suite struct that absorbs both the basic suite functionality from testify via `suite.Suite` and the suite functionality from the Temporal test framework via `testsuite.WorkflowTestSuite` [cf4aa1182]. For testing Workflows, you use a `testsuite.TestWorkflowEnvironment`, and you implement a `SetupTest` method to set up a new test environment before each test, which ensures that each test runs in its own isolated sandbox [cf4aa1182]. An `AfterTest` function can assert that all the mocks you set up were indeed called by invoking `s.env.AssertExpectations(s.T())`, and a timeout for the entire test can be set using `SetTestTimeout` in the Workflow or Activity environment [cf4aa1182].

### Testing and mocking Activities

An Activity can be tested with a mock Activity environment, which provides a way to mock the Activity context, listen to Heartbeats, and cancel the Activity; this lets you test the Activity in isolation by calling it directly, without needing to create a Worker to run the Activity [cf4aa1182]. The test framework provides two mechanisms for testing Workflow logic in isolation: Activity mocking and Activity overriding, both of which allow you to change the behavior of Activities invoked by your Workflow without the need to modify the actual Workflow code [cf4aa1182]. With Activity mocking, you set up a mock on the test environment via `s.env.OnActivity(...)` to return a desired value or error [cf4aa1182]. When you instead provide a function implementation as the parameter to `Return`, the framework executes this function whenever the Activity is invoked and passes on the return value as the result of the Activity invocation, and the framework will validate that the signature of the mock function matches the signature of the original Activity function [cf4aa1182]. Note that unless the Activity invocations are mocked or the Activity implementation replaced, the test environment will execute the actual Activity code including any calls to outside services [cf4aa1182]. When integration testing Workflows with a Worker, you can mock Activities by providing mock Activity implementations to the Worker [cf4aa1182].

### Executing and querying Workflows

Calling `s.env.ExecuteWorkflow(...)` executes the Workflow logic and any invoked Activities inside the test process, where the first parameter contains the Workflow function and any subsequent parameters contain values for custom input parameters declared by the Workflow [cf4aa1182]. After executing the Workflow, you assert that the Workflow ran through completion via `s.env.IsWorkflowCompleted()` and that no errors were returned via `s.env.GetWorkflowError()`, and if the Workflow returned a value you can retrieve it via `s.env.GetWorkflowResult(&value)` [cf4aa1182]. `TestWorkflowEnvironment` instances have a `QueryWorkflow()` method that lets you query the state of the currently running Workflow, but you should always query the Workflow either after `ExecuteWorkflow()` is done or in a `RegisterDelayedCallback()` callback, otherwise you'll get a `runtime error` panic [cf4aa1182].

### Time-skipping

Some long-running Workflows can persist for months or even years, and implementing the test framework allows your Workflow code to skip time and complete your tests in seconds rather than the Workflow's specified amount [cf4aa1182]. The test framework included in most SDKs is an in-memory implementation of Temporal Server that supports skipping time [cf4aa1182]. Time is a global property of an instance of `TestWorkflowEnvironment`: skipping time, either automatically or manually, applies to all currently running tests, so if you need different time behaviors for different tests, run your tests in a series or with separate instances of the test server [cf4aa1182]. In the time-skipping mode, Timers, which include sleeps and conditional timeouts, are fast-forwarded except when Activities are running [cf4aa1182]. `RegisterDelayedCallback()` does not actually make your test wait — Temporal's test framework advances time internally — and it can also be used to send Signals [cf4aa1182].

### Replaying Workflow Executions

Replay recreates the exact state of a Workflow Execution from the beginning of its Event History, and succeeds only if the Workflow Definition is compatible with the provided history from a deterministic point of view [cf4aa1182]. As part of CI checks when testing changes to your Workflow Definitions, the recommendation is to determine which Workflow Types or Task Queues will be targeted, download the Event Histories of a representative set of recent open and closed Workflows from each Task Queue, run the Event Histories through replay, and fail CI if any error is encountered during replay [cf4aa1182]. You use the `worker.WorkflowReplayer` to replay an existing Workflow Execution from its Event History to replicate errors, registering the Workflow function and calling `ReplayWorkflowHistory`; if a noticeably different code path was followed or some code caused a deadlock, it will be returned in the error code [cf4aa1182].
