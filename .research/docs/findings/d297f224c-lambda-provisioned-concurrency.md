# Lambda provisioned concurrency

## What it is

In Lambda, concurrency is "the number of in-flight requests that your function is currently handling," and there are "two types of concurrency controls available" [ca2f1f62b]. Provisioned concurrency is "the number of pre-initialized execution environments allocated to your function," and "these execution environments are ready to respond immediately to incoming function requests" [ca2f1f62b]. It "is useful for reducing cold start latencies for functions and designed to make functions available with double-digit millisecond response times" [ca2f1f62b]. A function using provisioned concurrency "does not exhibit cold start behavior since the execution environment is prepared ahead of invocation" [ca2f1f62b].

"Generally, interactive workloads benefit the most from the feature," namely "applications with users initiating requests, such as web and mobile applications," which "are the most sensitive to latency," whereas "asynchronous workloads, such as data processing pipelines, are often less latency sensitive and so do not usually need provisioned concurrency" [ca2f1f62b].

## How it differs from reserved concurrency

Reserved concurrency "sets both the maximum and minimum number of concurrent instances allocated to your function," and "when a function has reserved concurrency, no other function can use that concurrency" [ca2f1f62b]. Reserved concurrency "acts as both a lower and upper bound - it reserves the specified capacity exclusively for your function while also preventing it from scaling beyond that limit," and "configuring reserved concurrency for a function incurs no additional charges" [ca2f1f62b]. By contrast, "configuring provisioned concurrency incurs additional charges to your AWS account" [ca2f1f62b]. The two can coexist on the same function: "it's possible to allocate both reserved concurrency and provisioned concurrency for the same function," and "in such cases, the provisioned concurrency cannot exceed the reserved concurrency" [ca2f1f62b]. This limitation "extends to function versions," where "the maximum provisioned concurrency you can assign to a specific function version is the function's reserved concurrency minus the provisioned concurrency on other function versions" [ca2f1f62b].

## How you configure it

"You can configure provisioned concurrency settings for a function using the Lambda console or the Lambda API" [ca2f1f62b]. In the console, you open the Functions page, choose the function, choose "Configuration" then "Concurrency," and under "Provisioned concurrency configurations" choose "Add configuration," then choose the qualifier type and alias or version, enter a number under "Provisioned concurrency," and choose "Save" [ca2f1f62b]. "You cannot use provisioned concurrency with the $LATEST version of any function," and "if your function has an event source, make sure that event source points to the correct function alias or version," otherwise "your function won't use provisioned concurrency environments" [ca2f1f62b].

There is an account-level ceiling on how much you can allocate: "you can configure up to the Unreserved account concurrency in your account, minus 100," because "the remaining 100 units of concurrency are for functions that aren't using reserved concurrency" [ca2f1f62b]. The doc's example states that "if your account has a concurrency limit of 1,000, and you haven't assigned any reserved or provisioned concurrency to any of your other functions, you can configure a maximum of 900 provisioned concurrency units for a single function" [ca2f1f62b].

To configure provisioned concurrency with the Lambda API, the doc lists the `PutProvisionedConcurrencyConfig`, `GetProvisionedConcurrencyConfig`, `ListProvisionedConcurrencyConfigs`, and `DeleteProvisionedConcurrencyConfig` API operations [ca2f1f62b]. With the AWS CLI you use the `put-provisioned-concurrency-config` command; the doc's example allocates "100 units of provisioned concurrency for the BLUE alias of a function named my-function" [ca2f1f62b]. The example response shows fields including `"Requested ProvisionedConcurrentExecutions": 100`, `"Allocated ProvisionedConcurrentExecutions": 0`, and `"Status": "IN_PROGRESS"` [ca2f1f62b].

## Estimating how much you need

"You can view any active function's concurrency metrics using CloudWatch metrics," and "specifically, the ConcurrentExecutions metric shows you the number of concurrent invocations for functions in your account" [ca2f1f62b]. The doc's worked example reads a graph where "this function serves an average of 5 to 10 concurrent requests at any given time, and peaks at 20 requests," and advises that "if this function is critical to your application and you need a low-latency response on every invocation, configure at least 20 units of provisioned concurrency" [ca2f1f62b].

You can also calculate concurrency with the formula `Concurrency = (average requests per second) * (average request duration in seconds)` [ca2f1f62b]. "To estimate how much concurrency you need, multiply average requests per second with the average request duration in seconds," where you "estimate average requests per second using the Invocation metric, and the average request duration in seconds using the Duration metric" [ca2f1f62b]. "When configuring provisioned concurrency, Lambda suggests adding a 10% buffer on top of the amount of concurrency your function typically needs"; the doc's example is that "if your function usually peaks at 200 concurrent requests, set the provisioned concurrency to 220 (200 concurrent requests + 10% = 220 provisioned concurrency)" [ca2f1f62b].

## Code optimization when using it

"If you're using provisioned concurrency, consider restructuring your function code to optimize for low latency" [ca2f1f62b]. "For functions using provisioned concurrency, Lambda runs any initialization code, such as loading libraries and instantiating clients, during allocation time," so "it's advisable to move as much initialization outside of the main function handler to avoid impacting latency during actual function invocations" [ca2f1f62b]. In contrast, "initializing libraries or instantiating clients within your main handler code means your function must run this each time it's invoked (this occurs regardless of whether you're using provisioned concurrency)" [ca2f1f62b].

For on-demand invocations, "Lambda may need to rerun your initialization code every time your function experiences a cold start," and for such functions "you may choose to defer initialization of a specific capability until your function needs it" [ca2f1f62b]. The doc gives an example of initializing a client inside an `if` statement so that "Lambda runs this code only if some_condition is met," whereas initializing it outside the main handler means "Lambda runs that code on every cold start," which "can increase overall latency" [ca2f1f62b]. "You can measure cold starts as Lambda scales up by adding X-Ray monitoring to your function" [ca2f1f62b]. Because "provisioned concurrency must be applied to a specific version or alias of a function, not the $LATEST version," then "in cases where you continue to see cold start behavior, ensure that you are invoking the version of alias with provisioned concurrency configured" [ca2f1f62b].

The doc also notes a runtime-specific control: "if you're using the .NET 8 runtime, you can configure the AWS_LAMBDA_DOTNET_PREJIT environment variable to improve the latency for functions, even if they don't use provisioned concurrency," with the default value `ProvisionedConcurrency` meaning "Lambda performs ahead-of-time JIT compilation for all environments using provisioned concurrency" [ca2f1f62b].

## Interaction with scaling and cost

Configuring provisioned concurrency draws from the shared pool: "configuring provisioned concurrency for a function has an impact on the concurrency pool available to other functions," so that "if you configure 100 units of provisioned concurrency for function-a, other functions in your account must share the remaining 900 units of concurrency," and "this is true even if function-a doesn't use all 100 units" [ca2f1f62b].

When provisioned concurrency runs out, "it's possible for your function to use up all of its provisioned concurrency," and "Lambda uses on-demand instances to handle any excess traffic" [ca2f1f62b]. You can tell which type was used for a given environment by checking the `AWS_LAMBDA_INITIALIZATION_TYPE` environment variable, which "has two possible values: provisioned-concurrency or on-demand"; its value "is immutable and remains constant throughout the lifetime of the environment" [ca2f1f62b].

On cost and logging: "for provisioned concurrency environments, your function's initialization code runs during allocation, and periodically as Lambda recycles instances of your environment," and "Lambda bills you for initialization even if the environment instance never processes a request" [ca2f1f62b]. "Provisioned concurrency runs continually and incurs separate billing from initialization and invocation costs" [ca2f1f62b]. Lambda "logs the Init Duration field of the function in a platform-initReport log event in JSON logging format every time the environment is initialized," and to see it you "configure your JSON log level to at least INFO" [ca2f1f62b].

You can automate scaling with Application Auto Scaling "to manage provisioned concurrency on a schedule or based on utilization": "if your function receives predictable traffic patterns, use scheduled scaling," and "if you want your function to maintain a specific utilization percentage, use a target tracking scaling policy" [ca2f1f62b]. The doc warns that you must "configure an initial provisioned concurrency value first," because "if your function doesn't have an initial provisioned concurrency value, Application Auto Scaling may not handle function scaling properly" [ca2f1f62b]. With target tracking, "Application Auto Scaling creates and manages a set of CloudWatch alarms," and "when these alarms activate, Application Auto Scaling automatically adjusts the amount of environments allocated using provisioned concurrency" [ca2f1f62b]. The doc's example keeps "utilization near 70 percent," noting "you can apply any value between 10% and 90%"; the resulting high alarm "triggers when the utilization of provisioned concurrency consistently exceeds 70%" and the low alarm "triggers when utilization is consistently less than 63% (90 percent of the 70% target)" [ca2f1f62b].

When demand grows, "Application Auto Scaling increases provisioned concurrency in large steps until it reaches the configured maximum," after which "the function can continue to scale on standard, unreserved concurrency if you haven't reached your account concurrency limit," and "when utilization drops and stays low, Application Auto Scaling decreases provisioned concurrency in smaller periodic steps" [ca2f1f62b]. For bursty traffic, "Application Auto Scaling requires the burst load to sustain for at least 3 minutes in order to provision additional environments," and "both CloudWatch alarms require 3 data points that hit the target average to activate the auto scaling policy," so for functions experiencing quick bursts "using the Maximum statistic instead of the Average statistic can be more effective at scaling provisioned concurrency to minimize cold starts" [ca2f1f62b]. A billing caveat applies during idle periods: "Lambda emits the ProvisionedConcurrencyUtilization metric only when your function is active and receiving requests," and "during periods of inactivity, no metrics are emitted, and your auto-scaling alarms will enter the INSUFFICIENT_DATA state," so "Application Auto Scaling won't be able to adjust your function's provisioned concurrency," which "may lead to unexpected billing" [ca2f1f62b].
