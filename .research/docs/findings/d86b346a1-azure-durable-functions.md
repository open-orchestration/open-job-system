# Azure Durable Functions: Durable Execution Model

Durable Functions is an extension of Azure Functions that adds stateful orchestration to a function app, with each function playing one of four roles: orchestrator, activity, entity, or client [ce7a6a231]. Together these roles provide a durable, replay-based execution engine for reliable long-running workflows defined entirely in code [cfa15355b].

## Orchestration and replay

A durable orchestration uses an orchestrator function to coordinate the execution of other functions in a reliable, long-running workflow defined in procedural code, with no declarative schemas or designers [cfa15355b]. Orchestrator functions automatically checkpoint execution progress when the function calls an `await` or `yield` operator, so local state isn't lost when the process recycles or the VM reboots [cfa15355b]. An orchestration instance's lifespan can be seconds, days, or months, or it can be configured never to end [cfa15355b].

Orchestrator functions use the event-sourcing design pattern to maintain execution state: rather than storing the current state directly, the Durable Task Framework uses an append-only store to record the full series of actions the orchestration takes [cfa15355b]. Behind the scenes, the `await` operator in C# (and `yield` in JavaScript and Python) yields control of the orchestrator thread back to the Durable Task Framework dispatcher [cfa15355b]. The dispatcher then commits any new actions the orchestrator schedules to storage — appending all new events to the execution history like an append-only log, and creating messages in storage to schedule the actual work — after which the orchestrator function can be unloaded from memory [cfa15355b]. By default, Durable Functions uses Azure Storage as its runtime state store, though other storage providers are supported [cfa15355b].

When an orchestration gets more work to do (for example, a response message arrives or a durable timer expires), the orchestrator wakes up and re-executes the entire function from the start to rebuild its local state [cfa15355b]. During this replay, when the code tries to call a function, the framework consults the execution history; if it finds that the activity already executed and yielded a result, it replays that result and the orchestrator continues, repeating until the function finishes or schedules new asynchronous work [cfa15355b]. For replay to work correctly and reliably, orchestrator function code must be deterministic, and nondeterministic code can cause runtime errors or other unexpected behavior [cfa15355b]. At each checkpoint the framework saves the execution history into durable storage, enqueues messages for the functions the orchestrator wants to invoke, and enqueues messages for the orchestrator itself such as durable timer messages [cfa15355b]. The saved execution history is referred to as the orchestration history and is stored in a Table Storage history table keyed by the instance ID [cfa15355b].

## Task hubs

A task hub is a representation of the current state of the application in storage, including all pending work; while the application runs, the task hub continually stores the progress of orchestration, activity, and entity functions so the application can resume where it left off after a restart [c15a8c921]. Conceptually a task hub stores the instance states of all orchestration and entity instances, the activity messages representing activities waiting to run, and the instance messages waiting to be delivered to instances [c15a8c921]. Activity messages are stateless and can be processed anywhere, whereas instance messages must be delivered to a particular stateful instance identified by its instance ID [c15a8c921]. A running application continuously fetches work items from the task hub — activity work items run an activity function, and orchestrator work items run an orchestrator or entity function to process one or more instance messages [c15a8c921].

Each storage provider organizes the task hub differently [c15a8c921]. With the Azure Storage provider, two Azure Tables store the instance states, one Azure Queue stores the activity messages, and one or more control queues store the instance messages — each control queue representing a partition that is assigned a subset of all instance messages based on the hash of the instance ID — plus a few extra blob containers for lease blobs or large messages [c15a8c921]. Each application that shares a backend should connect to its own task hub, because apps sharing one task hub compete for messages and can produce undefined behavior such as orchestrations getting unexpectedly stuck [c15a8c921].

## Application components and patterns

Orchestrators define the workflow — what actions to take, in what order, and how to handle results — written as ordinary code using standard loops, conditionals, and try/catch blocks [ce7a6a231]. An orchestrator can schedule activities, sub-orchestrations, durable timers, and external events [ce7a6a231]. Orchestrator code must be deterministic, because the runtime uses event sourcing and replay to rebuild orchestrator state [ce7a6a231].

Activities are the basic unit of work in a durable orchestration, each typically representing a single task such as calling a web API, writing to a database, or computing a result, and orchestrators call activities to do their real work [ce7a6a231]. Unlike orchestrators, activities have no code restrictions and can run any code, including nondeterministic or long-running operations [ce7a6a231]. The runtime guarantees that each activity runs at least once, and if a failure occurs after an activity completes but before its result is recorded the runtime might rerun it, so activity logic should be idempotent where possible [ce7a6a231]. Each activity receives one input and returns one output, and activities can run serially, in parallel, or in a combination of both [ce7a6a231]. The client component starts and manages orchestrations and entities from outside an orchestration, including scheduling new instances, querying status, raising events, suspending and resuming, and terminating instances [ce7a6a231].

## Durable entities

Entities manage small, durable pieces of state, acting like small services that communicate by using messages, with each entity having a unique identity and, if needed, internal state [cf370497c]. Entities communicate with other entities, orchestrations, and clients through messages that the runtime sends through reliable queues [cf370497c]. To prevent conflicts, a single entity runs operations serially, one after another [cf370497c]. An entity is addressed by an entity ID, a pair of strings — an entity name identifying the entity type and an entity key uniquely identifying the instance among entities of the same name — and to invoke an operation you specify the target entity ID, an operation name, an optional input, and an optional scheduled time [cf370497c]. Operations can return a result value or an error result, and an operation can create, read, update, and delete the entity state, which the runtime always persists in storage [cf370497c]. To coordinate updates across multiple entities, an orchestrator can use the .NET `LockAsync` method to create a critical section; locks are durable and persist even if the executing process is recycled, and other clients can't run operations on a locked entity until the holding orchestration releases the lock [cf370497c].

## Eternal orchestrations

Eternal orchestrations run indefinitely by periodically resetting their own history using the `continue-as-new` API, and are useful for aggregators, periodic background jobs, and any scenario needing an infinite loop [c987ab833]. Without `continue-as-new`, an orchestration that loops forever would accumulate history with every scheduled task, eventually causing performance problems and excessive memory use, which the pattern solves by resetting the history on each iteration [c987ab833]. The `continue-as-new` method takes a JSON-serializable parameter that becomes the new input for the next generation; the instance restarts itself, keeping the same instance ID while the orchestration's history resets [c987ab833]. The Durable Task Framework keeps the same instance ID but internally creates a new execution ID going forward, and the results of any incomplete tasks are discarded when an orchestration calls `continue-as-new` [c987ab833]. An orchestration in an infinite loop that needs to be stopped is halted using the terminate API of the orchestration client [c987ab833].

## Error handling

Because orchestrations are implemented in code, error handling uses the language's built-in features, and the Durable Task SDKs add support for error handling, retries, and timeouts [c986ff4cd]. Unhandled exceptions thrown within activity functions or sub-orchestrations are marshaled back to the orchestrator function using standardized exception types [c986ff4cd]. When calling activity or sub-orchestration functions, an orchestrator can specify an automatic retry policy whose options are the maximum number of attempts, the first retry interval, a backoff coefficient (defaulting to 1), the maximum retry interval, and a retry timeout [c986ff4cd]. To bound how long a call can take, an orchestrator times it out by creating a durable timer and racing it against the activity task with an `any`-style selector [c986ff4cd]. Compensation is written explicitly: when a later call fails, the orchestrator catches the exception and compensates, for example by crediting funds back to the source account [c986ff4cd]. If an orchestrator function fails with an unhandled exception, the runtime logs the exception details and the instance completes with a `Failed` status [c986ff4cd]. Consistent with this, an orchestration instance that finishes in a `Failed` state from an unhandled exception can't be retried [cfa15355b].

## Versioning of in-flight orchestrations

Versioning matters because functions are inevitably added, removed, and changed over an application's lifetime, and these changes affect both new and existing orchestrations [c1040e934]. A signature change — altering the name, input, or output of an activity or entity function — can break in-flight instances: if an instance checkpointed a Boolean result and the new code expects a String, the checkpointed instance fails when it replays the call because the history holds the old type [c1040e934]. Likewise, changing orchestrator logic to add, remove, or reorder calls changes the execution path, so during replay the runtime detects the inconsistency and raises a non-deterministic orchestration error [c1040e934]. The recommended mitigation is orchestration versioning, a built-in runtime feature in which each instance gets a version permanently associated with it at creation, orchestrators can branch on their version, workers running newer versions can continue executing instances created by older versions, and the runtime prevents older workers from executing newer instances [c1040e934]. Alternative strategies are stopping all in-flight instances (acceptable mainly for prototyping and local development) and side-by-side deployments using a different storage account or a different task hub name, optionally swapped in via deployment slots [c1040e934].
