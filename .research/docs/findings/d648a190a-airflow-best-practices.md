# Apache Airflow DAG authoring best practices

Creating a new DAG is a three-step process: writing Python code to create a DAG object, testing if the code meets expectations, and configuring environment dependencies to run the DAG [ce9d10d36]. There are many things to take care of to ensure a DAG run or failure does not produce unexpected results [ce9d10d36].

## Deterministic, retry-safe tasks

Tasks in Airflow should be treated as equivalent to transactions in a database, meaning a task should never produce incomplete results — for example, it should not produce incomplete data in `HDFS` or `S3` at the end of a task [ce9d10d36]. Because Airflow can retry a task if it fails, the tasks should produce the same outcome on every re-run [ce9d10d36]. The doc lists ways to avoid producing a different result: do not use INSERT during a task re-run because an INSERT statement might lead to duplicate rows, and replace it with UPSERT instead [ce9d10d36]. It also advises reading and writing in a specific partition and never reading the latest available data in a task, since someone may update the input data between re-runs and produce different outputs; `data_interval_start` can be used as a partition, and the same partitioning method should be followed when writing data to S3/HDFS [ce9d10d36]. The Python datetime `now()` function gives the current datetime object and should never be used inside a task — especially for critical computation — because it leads to different outcomes on each run, though it is fine to use it, for example, to generate a temporary log [ce9d10d36].

The doc adds a tip that repetitive parameters such as `connection_id` or S3 paths should be defined in `default_args` rather than declared for each task, which helps avoid mistakes such as typographical errors [ce9d10d36].

## Custom operators and hooks

For writing a custom operator or hook, the best-practices page directs the reader to follow the Airflow guide on custom Operators rather than restating its details inline [ce9d10d36].

## Avoiding top-level code in DAG files

You should avoid writing top-level code that is not necessary to create Operators and build DAG relations between them, because of the design decision for the Airflow scheduler and the impact of top-level code parsing speed on both performance and scalability of Airflow [ce9d10d36]. The Airflow scheduler executes the code outside the Operator's `execute` methods with the minimum interval of `min_file_process_interval` seconds, in order to allow dynamic scheduling of the DAGs — where scheduling and dependencies might change over time and impact the next schedule of the DAG — and the scheduler tries to continuously make sure that what is in DAGs is correctly reflected in scheduled tasks [ce9d10d36]. Specifically, you should not run any database access, heavy computations, or networking operations at the top level [ce9d10d36].

One important factor impacting DAG loading time, which might be overlooked by Python developers, is that top-level imports might take surprisingly a lot of time and generate a lot of overhead, and this can easily be avoided by converting them to local imports inside Python callables [ce9d10d36]. The doc states that in its two examples, the DAG that does not avoid top-level code will take an additional 1000 seconds to parse compared to the functionally equivalent DAG where the `expensive_api_call` is executed from the context of its task [ce9d10d36]. In the first example the expensive call is executed each time the DAG file is parsed, resulting in suboptimal performance in the DAG file processing, whereas in the second example it is only called when the task is running and thus can be parsed without suffering any performance hits [ce9d10d36]. Import statements also count as top-level code, so an import that takes a long time, or an imported module that itself executes code at the top level, can also impact the performance of the scheduler — the doc shows moving expensive imports such as `pandas`, `torch`, and `tensorflow` into the task functions [ce9d10d36].

To check whether a piece of code is "top-level" code, the doc notes that Python executes the code it sees when it parses a file, except (in general) the internal code of methods it does not execute, and that there are non-obvious special cases — for example, top-level code also means any code used to determine default values of methods [ce9d10d36]. The practical test it gives is to add print statements to the code in question and run `python <my_dag_file>.py` to see which pieces actually execute during parsing [ce9d10d36].

Related guidance: using Airflow Variables yields network calls and database access, so their usage in top-level Python code for DAGs should be avoided as much as possible [ce9d10d36]. Likewise, the Timetables section advises avoiding the use of Airflow Variables/Connections or accessing the Airflow database at the top level of timetable code, and delaying database access until the execution time of the DAG [ce9d10d36].

## Communicating between tasks (XCom)

Airflow executes the tasks of a DAG on different servers when using the Kubernetes executor or Celery executor, so you should not store any file or config in the local filesystem, as the next task is likely to run on a different server without access to it [ce9d10d36]. Even with the `Local executor`, storing a file on disk can make retries harder, for example when a task requires a config file that is deleted by another task in the DAG [ce9d10d36]. Where possible, use `XCom` to communicate small messages between tasks, and a good way of passing larger data between tasks is to use remote storage such as S3/HDFS — for instance, a task that stores processed data in S3 can push the S3 path into `XCom`, and downstream tasks can pull the path from XCom and use it to read the data [ce9d10d36]. Tasks should also not store authentication parameters such as passwords or tokens inside them; where possible, Connections should be used to store such data securely in the Airflow backend and retrieve it using a unique connection id [ce9d10d36].

## Deleting tasks and pausing DAGs

The doc warns to be careful when deleting a task from a DAG, because you would not be able to see the task in Graph View, Grid View, etc., making it difficult to check that task's logs from the Webserver; if that is not desired, a new DAG should be created instead [ce9d10d36]. For maintenance windows, the doc recommends the `dags pause` command as a better (though more manual) way to stop scheduling: you need to keep track of which DAGs are paused before you begin so you know which to unpause afterward, by first running `airflow dags list` to store the list of unpaused DAGs and then using that same list to run both `dags pause` for each DAG prior to maintenance and `dags unpause` after [ce9d10d36]. A benefit it notes is that you can try un-pausing just one or two DAGs, perhaps dedicated test DAGs, to validate before resuming the rest [ce9d10d36].

## Testing DAGs

Airflow users should treat DAGs as production-level code, and DAGs should have various associated tests to ensure they produce expected results [ce9d10d36]. The DAG Loader Test ensures a DAG does not contain code that raises an error while loading, requires no additional code to be written by the user, and is run simply with `python your-dag-file.py`; running that command without error ensures the DAG contains no uninstalled dependency or syntax errors, and it should be loaded in an environment that corresponds to the scheduler environment with the same dependencies, environment variables, and common code [ce9d10d36]. The same load test is described as a way to check whether a DAG loads faster after optimization, using the built-in `time` command and running it several times in succession to account for caching effects, comparing results before and after the optimization under the same conditions [ce9d10d36]. The doc points out that the "real time" metric tells you how long the DAG took to process, and that loading the file this way starts a new interpreter so there is an initial loading time not present when Airflow parses the DAG [ce9d10d36].

Unit tests ensure there is no incorrect code in a DAG, and can be written for both tasks and the DAG itself, including a unit test for loading a DAG via `DagBag` [ce9d10d36]. Self-Checks can be implemented inside a DAG to verify that tasks are producing expected results — for example, a task that pushes data to S3 can be followed by a check (such as an `S3KeySensor`) that confirms the partition was created and performs simple checks on the data [ce9d10d36]. The doc also recommends keeping a staging environment, if possible, to test the complete DAG run before deploying to production, and parameterizing the DAG (for example via environment variables) to change values such as the output path or database rather than hard-coding values and changing them manually per environment [ce9d10d36].

## Reducing DAG complexity and performance guidance

While Airflow handles many DAGs with many tasks and dependencies, the complexity of many complex DAGs might impact scheduling performance, so you should strive to simplify and optimize DAGs whenever possible — DAG parsing and creation is just executing Python code, and it is up to the DAG writer to make it as performant as possible [ce9d10d36]. The doc states there are no metrics for DAG complexity that tell you whether a DAG is "simple enough," but offers concrete actions [ce9d10d36]. The single improvement with the biggest impact on scheduler performance is making the DAG load faster, which ties back to the top-level-code guidance and the DAG Loader Test for assessing loading time [ce9d10d36]. It also advises making the DAG generate a simpler structure, because every task dependency adds processing overhead for scheduling and execution — a simple linear structure `A -> B -> C` experiences less delay in task scheduling than a deeply nested tree with an exponentially growing number of dependent tasks, so making DAGs more linear is preferred [ce9d10d36].

For dynamic DAG generation, the doc says avoiding excessive processing at the top-level code is especially important, and that dynamic DAG configuration can be supplied via environment variables, via externally provided generated Python code containing meta-data in the DAG folder, or via an externally provided generated configuration meta-data file in the DAG folder [ce9d10d36]. On metadata-database resources, it notes the metadata database increases its storage footprint over time as more DAG and task runs and event logs accumulate, and that the `airflow db clean` CLI command can be used to purge old data [ce9d10d36].
