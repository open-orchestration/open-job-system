# Kubernetes Batch Jobs: Reliability and Parallelism

The Kubernetes `batch/v1` Job object represents the configuration of a single job [ce572728a]. It runs one or more Pods to completion and tracks success through two coupled `JobSpec` fields. `completions` specifies the desired number of successfully finished Pods the Job should run with; setting it to null means the success of any Pod signals the success of all Pods and lets `parallelism` take any positive value, while setting it to 1 limits `parallelism` to 1 so that the success of that single Pod signals the success of the Job [ce572728a]. `parallelism` specifies the maximum desired number of Pods the Job should run at any given time, and the actual number running in steady state is lower when the work left to do (`.spec.completions - .status.successful`) is less than that maximum [ce572728a]. The controller exposes progress through `JobStatus`: `active` counts pending and running non-terminating Pods, `succeeded` counts Pods that reached phase Succeeded, and `failed` counts Pods that reached phase Failed, with `succeeded` and `failed` increasing monotonically [ce572728a].

## Completion modes: Indexed vs NonIndexed

`completionMode` specifies how Pod completions are tracked and is either `NonIndexed` (the default) or `Indexed` [ce572728a]. In `NonIndexed` mode the Job is considered complete when there have been `.spec.completions` successfully completed Pods, and each Pod completion is homologous to every other [ce572728a]. In `Indexed` mode the Pods of a Job get an associated completion index from 0 to `.spec.completions - 1`, available in the annotation `batch.kubernetes.io/job-completion-index`, and the Job is considered complete when there is one successfully completed Pod for each index [ce572728a]. When the value is `Indexed`, `.spec.completions` must be specified and `.spec.parallelism` must be less than or equal to 10^5 [ce572728a]. `JobStatus.completedIndexes` records the completed indexes, in increasing order, when `completionMode = "Indexed"` [ce572728a].

Indexed Jobs implement static work assignment: multiple parallel workers each run as a separate container in their own Pod, and each Pod carries an index number that the control plane sets automatically so it can identify which part of the overall task to work on [cb7c4e028]. The pod index lives in the `batch.kubernetes.io/job-completion-index` annotation, and for convenience the control plane automatically wires the downward API to expose it in the `JOB_COMPLETION_INDEX` environment variable [cb7c4e028]. When such a Job is created, the control plane creates one Pod per index; `.spec.parallelism` determines how many can run at once whereas `.spec.completions` determines how many Pods the Job creates in total, and because `.spec.parallelism` is less than `.spec.completions` the control plane waits for some of the first Pods to complete before starting more [cb7c4e028].

## Work-queue (fine parallel) coordination

In the fine parallel work-queue pattern, the Job runs multiple parallel tasks as worker processes, each running as a separate Pod, and as each Pod is created it picks up one unit of work from a task queue, processes it, and repeats until the end of the queue is reached [c21dc74a2]. The pattern uses an external storage service (the example uses Redis) to hold the work queue, which is filled with messages where each message represents one task; the Job then starts several Pods, and each Pod takes one task from the message queue, processes it, and repeats until the queue is exhausted [c21dc74a2]. Coordination is left to the workers rather than the Job controller: because the workers themselves detect when the work queue is empty and the Job controller does not know about the queue, the controller relies on the workers to signal that they are done by exiting with success, so that as soon as any worker exits with success the controller knows the work is done [c21dc74a2]. For this reason the example leaves the Job's completion count unset, and the Job controller then waits for the other Pods to complete too; the manifest sets `parallelism: 2` [c21dc74a2].

## Reliability: Pod failure policy and the backoff limit

The first line of failure handling is the backoff limit. `backoffLimit` specifies the number of retries before marking a Job failed, defaulting to 6 unless `backoffLimitPerIndex` (only for Indexed Jobs) is specified [ce572728a]. `podFailurePolicy` specifies the policy for handling failed Pods, allowing a set of actions and conditions to be defined; if it is empty the default behaviour applies, where the counter of failed Pods represented by the Job's `.status.failed` field is incremented and checked against the `backoffLimit`, and the field cannot be used in combination with `restartPolicy=OnFailure` [ce572728a]. A `PodFailurePolicy` describes how failed Pods influence the `backoffLimit` [ce572728a]. Each rule's `action` selects what happens when its requirements are satisfied: `FailJob` marks the Pod's Job as Failed and terminates all running Pods; `FailIndex` marks the Pod's index as Failed and does not restart it; `Ignore` does not increment the counter toward `.backoffLimit` and creates a replacement Pod; and `Count` handles the Pod in the default way by incrementing the counter toward `.backoffLimit` [ce572728a].

The task guidance frames this as combining a Pod failure policy with the default Pod backoff failure policy to improve control over handling container- or Pod-level failures, which helps better utilize computational resources by avoiding unnecessary Pod retries and helps avoid Job failures due to Pod disruptions such as preemption, API-initiated eviction, or taint-based eviction [c8ef34c71]. A `FailJob` rule lets a Job avoid unnecessary Pod restarts when a Pod failure indicates a non-retriable software error: in the worked example the Job gains a `FailureTarget` condition with `reason` set to `PodFailurePolicy`, reporting that a container failed with exit code 42 matching the `FailJob` rule at index 0 [c8ef34c71]. An `Ignore` rule lets a Job keep Pod disruptions from incrementing the Pod retry counter, so that after a deliberately triggered disruption the Job's `.status.failed` counter is not incremented; the guidance notes that if the Pod failure policy were disabled, the same Pod disruption would terminate the entire Job because `.spec.backoffLimit` is set to 0 [c8ef34c71]. The motivation given is that, relying only on `.spec.backoffLimit`, it is often problematic to find a balance between a value low enough to avoid unnecessary Pod retries yet high enough that the Job is not terminated by Pod disruptions [c8ef34c71]. For per-index control, `backoffLimitPerIndex` specifies the retry limit within an index before marking that index failed and can only be set when `completionMode=Indexed` and the Pod's restart policy is `Never`, while `maxFailedIndexes` specifies the maximal number of failed indexes before the whole Job is marked Failed and its execution terminated [ce572728a].

## Disruptions and PodDisruptionBudgets

Reliability also depends on how running Pods survive disruptions. Involuntary disruptions are unavoidable cases such as a hardware failure of the physical machine backing the node, a cluster administrator deleting a VM by mistake, a cloud provider or hypervisor failure making a VM disappear, the node disappearing due to a cluster network partition, or eviction of a Pod because the node is out of resources [cdca2674b]. Voluntary disruptions include application-owner actions such as deleting the controller that manages the Pod, updating a Deployment's Pod template causing a restart, or directly deleting a Pod, as well as cluster-administrator actions such as draining a node for repair or upgrade, draining a node to scale the cluster down, or removing a Pod to permit something else to fit [cdca2674b]. An application owner can create a PodDisruptionBudget (PDB), which limits the number of Pods of a replicated application that are down simultaneously from voluntary disruptions [cdca2674b]. Cluster managers and hosting providers should use tools that respect PDBs by calling the Eviction API instead of directly deleting Pods, and `kubectl drain` marks a node out of service and tries to evict all Pods on it while respecting the configured budget [cdca2674b]. Not all voluntary disruptions are constrained by PDBs; for example, deleting Deployments or Pods bypasses PodDisruptionBudgets [cdca2674b]. A dedicated Pod `DisruptionTarget` condition indicates that a Pod is about to be deleted due to a disruption [cdca2674b].

## Scheduled execution with CronJob

Automated, recurring batch tasks are run with the `CronJob` object [cd7189fe4]. A CronJob is a `batch/v1` kind whose `spec.schedule` field carries a cron-syntax schedule — for example `"* * * * *"` runs a simple demonstration task every minute — and a `spec.jobTemplate` whose Pod template defines the work to run [cd7189fe4]. The CronJob schedules and creates Job objects at the specified time, and a successfully scheduled run shows up under the CronJob's `LAST SCHEDULE` with the resulting Job and its Pods [cd7189fe4]. Deleting the CronJob removes all the Jobs and Pods it created and stops it from creating additional Jobs [cd7189fe4].

## Lifecycle and cleanup controls

Several `JobSpec` fields govern the Job's runtime envelope and cleanup. `activeDeadlineSeconds` specifies the duration in seconds relative to `startTime` that the Job may be continuously active before the system tries to terminate it, and if a Job is suspended this timer is stopped and reset when the Job is resumed [ce572728a]. `suspend` specifies whether the Job controller should create Pods: if a Job is created with `suspend` true no Pods are created, and if a Job is suspended after creation the controller deletes all active Pods associated with it, while suspending also resets the Job's `StartTime` and therefore the `activeDeadlineSeconds` timer [ce572728a]. `ttlSecondsAfterFinished` limits the lifetime of a Job that has finished execution (either Complete or Failed): when set, the Job becomes eligible to be automatically deleted that many seconds after finishing, a value of zero makes it eligible immediately, and when unset the Job is not automatically deleted [ce572728a].
