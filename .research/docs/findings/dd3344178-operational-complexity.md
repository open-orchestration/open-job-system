**Scope.** This finding compares the five systems on operational / total-cost-of-ownership burden: the backing store(s) they require, the number of moving parts an operator must deploy and run, and the monitoring/UI story. It does not cover throughput, latency, or feature breadth except where those bear on what must be operated. Every one of the five entities is backed by at least one cited source. The dominant through-line is that **the number of distinct backing stores and long-running services drives operational complexity**, and the five systems span a wide range on that axis — from a single shared Postgres database (River) to a multi-process cluster plus a database and a visibility store (Temporal).

## The through-line: backing-store count drives op-complexity

### River — Postgres only, no separate broker (lightest)

River is "a robust high-performance job processing system for Go and Postgres" and "encourages the use of the same database for application data and job queue" by enqueueing jobs transactionally alongside other database changes [c685ed0ec]. It "requires an existing PostgreSQL database" and "persists jobs to a Postgres database, [needing] a small set of tables created to insert jobs and carry out leader election," which a bundled CLI migration tool (`river migrate-up`) provisions [ca2ca0f56]. The operational payoff is stated directly: River means "No extra infrastructure: Reuse your existing Postgres database, avoiding the need for additional services like Redis" [cc6ce16fe]. For observability River ships "a Web UI for inspecting and interacting with jobs and queues" [c685ed0ec], and job status can also be inspected directly in the `river_job` Postgres table [cc6ce16fe]. So River adds essentially zero new stores to a stack that already runs Postgres.

### Sidekiq and BullMQ — one extra store (Redis) plus a web UI

Both Sidekiq and BullMQ add exactly one backing store on top of the application: Redis.

For Sidekiq, Redis is the required store — "Redis 7.0+, Valkey 7.2+, or Dragonfly 1.27+" — and Sidekiq connects via a `REDIS_URL` environment variable [c983a4b42]. Multiple Sidekiq processes (the guidance is "1 or more Sidekiq processes per app server") all "[talk] to the same Redis server" [c33cbcb8c]. Monitoring is built in: Sidekiq "provides built-in monitoring through the web UI and API" [c983a4b42], the Web UI exposes overall system data at `/stats`, and Sidekiq 6.0 ships a `sidekiqmon` binary that prints stats to the terminal using `REDIS_URL` [c8e155433]. Heavier monitoring and process-management features are gated behind the commercial tiers — Sidekiq Enterprise's multi-process mode can "monitor memory usage and restart bloated worker processes" [c8e155433] and its `sidekiqswarm` binary starts N processes from a single systemd service [c33cbcb8c], while Sidekiq Pro is described as a "commercial extension with advanced monitoring" [c983a4b42].

For BullMQ, "Redis is an in-memory data structure store that BullMQ relies on for job state," and a production deployment must configure Redis durability — enable AOF (`appendonly yes`, `appendfsync everysec`) plus RDB snapshots — to avoid losing jobs on a crash [ce7433bff]. Monitoring is via a separate dashboard component: "Monitor with Bull Board for a real-time dashboard and dead-letter inspection" [ce7433bff]. (BullMQ here rests on a single promotional how-to source, so these claims are kept to exactly what it states: Redis as the job-state store, Redis-durability configuration, and Bull Board for monitoring.)

### Celery — broker, optional result backend, and Flower (more parts)

Celery's operational footprint is larger because it separates two roles. "Celery has the ability to communicate and store with many different backends (Result Stores) and brokers (Message Transports)" [c1d9724f7]. The broker is "a 'message transport' where created tasks are stored" and pulled from by workers [cf04262ad]; the result backend is a distinct concern that "tracks the outcomes and returned results of async tasks" [cf04262ad]. The result backend is optional: "Having a backend is not required for Celery to run tasks. However, it is required if you ever need to check the outcome of a task or return a task's result" [cf04262ad]. Redis can fill both roles ("Redis can be both a backend and a broker"), but the docs note that for durable results one may instead reach for PostgreSQL, MySQL, or Cassandra as the backend — i.e. potentially a second, different store [c1d9724f7]. Monitoring adds yet another component: Flower and other event-based monitoring tools work only when the chosen transport implements events ("Missing monitor support means that the transport doesn't implement events, and as such Flower, celery events, celerymon and other event-based monitoring tools won't work") [c1d9724f7], and enabling it requires `worker_send_task_events=True` on the worker [cd7ebc288]. A representative production setup runs Celery with Redis as both broker and result backend (isolating the result backend into "a separate Redis instance" with TTLs to cap memory) [cd7ebc288]. So a fully-featured Celery deployment is workers + a broker + (optionally) a result backend + a monitoring service — more moving parts than the single-Redis systems.

### Temporal — a full self-hosted cluster (heaviest)

Temporal carries the largest operational burden of the five. As one self-hosting operator puts it bluntly: "Temporal is not a single process with a single run command. It's a persistent database and a cluster of multiple processes" [c57c8b28f]. The official guide frames the self-hosted Temporal Service as "open source infrastructure software that orchestrates your durable applications" and, in self-hosted deployments, "a critical control and persistence component [that] should be secured similarly to a database" [c4a87ede8][c8ebc897b]. A reference docker-compose deployment "includes a PostgreSQL database, an Elasticsearch instance, and exposes the Temporal gRPC Frontend on port 7233," with the Web UI served separately at port 8080 [c8ebc897b]. The server and UI are distinct binaries — "a complete Temporal Server [is deployed] by deploying two Go binaries -- the core Temporal Server, and the Temporal UI Server" — and on Kubernetes the Helm charts deploy "the Temporal Server services to individual pods" connected to "your existing database and Elasticsearch instances" [c8ebc897b]. Real-world self-hosting confirms the parts list: an AWS deployment used RDS (Postgres, after starting on MySQL; Cassandra was also an option) for persistence, run as a cluster of services that must reach each other and the database over the network, with a chosen shard count [c57c8b28f]. Tooling exists to ease this — a deployment skill "simplifies the deployment and management of self-hosted Temporal clusters across Kubernetes, EKS, and local environments" and documents "production-ready PostgreSQL and Elasticsearch integration patterns" and "database schema initialization and migration" [c266bb52e] — but the underlying shape is still a multi-service cluster plus a database plus (typically) a visibility store, which is materially more to deploy, monitor, and upgrade than any of the other four. Temporal does offer a lighter on-ramp for non-production use: the Temporal CLI development server is "a single binary with no external dependencies" that starts a complete service with Web UI locally, explicitly distinguished from the production deployment path [c4a87ede8][c8ebc897b].

## Summary ordering (by backing-store / moving-part count)

From lightest to heaviest operational footprint, grounded in the sources above:

1. **River** — no new store beyond an existing Postgres; jobs and leader-election live in that same database; built-in Web UI [c685ed0ec][ca2ca0f56][cc6ce16fe].
2. **Sidekiq / BullMQ** — one extra store (Redis) plus a web UI/dashboard; Sidekiq's deeper monitoring and process management are commercial add-ons [c983a4b42][c8e155433][c33cbcb8c][ce7433bff].
3. **Celery** — a broker, an optional (and possibly separate) result backend, and an event-monitoring service (Flower) [c1d9724f7][cf04262ad][cd7ebc288].
4. **Temporal** — a cluster of multiple server processes, a persistence database, a visibility store (Elasticsearch), and a separate UI server [c57c8b28f][c8ebc897b][c4a87ede8][c266bb52e].
